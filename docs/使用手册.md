# 使用手册

## 标准流程

1. 把项目素材放入 `projects/<项目名称>/input/`。
2. 实际查看每张视觉素材，不只检查文件名、OCR、尺寸或元数据。
3. 在项目根目录的 `masterpiece-os.json` 记录 Brand Lock、视觉核验事实与 Original Intent。
4. 准备至少三个真正同类型、同定位且来源可追溯的 Benchmark；需要在线候选时使用 `--online`。
5. 记录行业观察、差异机会与 Creative Decision，再由 Creative Director 显式批准 Brand DNA。
6. 填写 Creative Reasoning 的其余九部分，运行分析并检查四份固定报告。
7. 对“待确认”内容补证后重新运行，直到 Brief 具备进入创意发展的基础。

## 运行项目

```bash
npm run analyze -- --project "项目名称" --online
```

只有一个本地项目时可以省略 `--project`。项目模式固定写入 `projects/<项目名称>/outputs/`。

直接分析素材目录：

```bash
npm run analyze -- "D:/项目/assets" --output "D:/项目/outputs" --config "D:/项目/masterpiece-os.json"
```

## 四份输出

- `01-项目分析报告.md`：确认素材证据、Brand Lock、Benchmark 与完整 Brand DNA Decision 链。
- `02-Creative-Brief.md`：面向品牌设计师和创意团队的十部分专业简报。
- `03-Knowledge-Review.md`：五类 Thinking Framework 问题与本项目待回答问题。
- `04-Design-Review.md`：检查十部分 Brief 的证据和准备状态。

`--debug` 会额外输出 `masterpiece-os-result.json`，用于开发排查。

## 配置重点

### visualInspection

```json
{
  "verified": true,
  "inspectedImageCount": 8,
  "inspectedImages": ["包装正面.jpg", "空间应用.jpg"],
  "findings": ["包装使用非对称网格", "摄影保留真实接触阴影"]
}
```

核验数量必须覆盖实际图片总数，否则系统会显示“视觉核验未闭环”。

### benchmarks

```json
{
  "benchmarks": [
    { "name": "案例 A", "url": "https://example.com/a", "reason": "同品类、同价格带，品牌系统完整" },
    { "name": "案例 B", "url": "https://example.com/b", "reason": "定位相近，可观察跨触点表达" },
    { "name": "案例 C", "url": "https://example.com/c", "reason": "摄影与材质语言具有参考价值" }
  ],
  "benchmarkResearch": {
    "verified": true,
    "status": "已人工联网核验 3 个案例"
  }
}
```

联网结果是候选，仍需人工核实是否真正同类型、同定位。不得把其他品牌特征写成本项目事实。

### brandDnaDecision

依次填写 `originalIntent`、`industryBenchmark`、`creativeDecision` 与 `approvedBrandDNA`。九个 DNA 维度完整后，由 Creative Director 把 `approval.status` 设为 `approved`。任一上游阶段不完整时，即使配置里已有 DNA 和批准标记，系统仍会阻止它进入 Brief。

旧 `creativeReasoning.visualDNA` 只作为迁移候选，不属于批准结论。

### creativeReasoning

填写 Brand Identity、Brand Positioning、Design Language、Emotional Direction、Photography Direction、Design Risks、Must Keep、Can Explore 与 Design Goal，完整字段见 `templates/masterpiece-os.json`。好的内容应说明设计关系和理由，而不是堆叠形容词。摄影部分描述长期方向，不写单张 Prompt。

## 判断报告是否合格

- 品牌名称、行业、Logo、色彩、字体、包装和核心资产与真实素材一致。
- 所有不能确认的信息明确标为“待确认”。
- Brand Identity 说明品牌真正是什么，而不只是销售品类。
- Design Language 解释视觉关系与原因，不只是关键词。
- Approved Brand DNA 能追溯到 Original Intent、Industry Benchmark 与 Creative Decision，并具有显式批准。
- Must Keep 和 Can Explore 同时存在，既保护识别也保留创意空间。
- Brief 不包含图片数量、画幅比例、任务编号或对 AI 的执行命令。
- Knowledge Review 只提出问题，不把项目答案升级为通用规则。

## 常见问题

### 为什么 Brief 仍有“待确认”？

因为视觉核验或品牌事实没有充分证据。补充实物、授权 Logo、色值、包装结构或逐张画面记录后更新配置并重跑。

### 为什么没有生成生图任务？

v3.2 将 Masterpiece-OS 的职责限定为品牌理解、Brand DNA Decision、Creative Brief 与 Thinking Framework。后续 GPT 以已核验视觉方案和 Brief 为输入，自主完成图片规划与生成。

### 旧的 `--mode review` 还能用吗？

可以作为兼容参数，但它和默认命令执行完全相同的 Creative Brief 工作流，不会恢复旧版图片规划或成长引擎。

### 会修改 Knowledge 或执行 Git 吗？

不会。分析命令只写项目输出，不改 Thinking Framework，不写成长历史，也不执行任何 Git 操作。
