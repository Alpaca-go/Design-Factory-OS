# 架构说明

Design Factory OS v3.1 的定位是 **AI Creative Brief Generator**。核心链路只有一条：

```text
Visuals → Brand Lock → Benchmark → Creative Reasoning → Creative Brief
```

## 模块

1. `project-initializer`：创建标准项目目录，并安全整理根目录素材。
2. `inventory` / `parsers`：盘点图片、文档、压缩包与演示文件的可解析元数据。
3. `analyze`：建立 Brand Lock，并组织项目类型、行业与同类对标语境。
4. `creative-reasoning`：把人工视觉核验和项目配置转换为十部分 Creative Brief 数据契约。
5. `thinking-framework`：只读五类设计思考问题，并生成项目需要继续回答的问题。
6. `brief-review`：检查 Brief 的证据、边界和进入创意发展的准备度。
7. `report`：固定渲染四份编号 Markdown；调试模式可额外写入 JSON。
8. `pipeline`：协调上述模块，不执行图片规划、Knowledge 写回、成长历史或 Git 操作。

## Creative Reasoning 数据契约

```text
brandIdentity
brandPositioning
designLanguage
emotionalDirection
visualDNA
photographyDirection
designRisks
mustKeep
canExplore
designGoal
```

配置可以显式提供这些字段。缺失字段只能从 Brand Lock、Benchmark 或已记录的视觉核验事实中安全回退；无法确认时保留“待确认”。

## 证据分层

```text
项目事实
├─ 实际素材与逐张视觉核验
├─ 项目配置中的人工确认
└─ 可追溯的公开来源

参考语境
└─ 同类 Benchmark（不得替代项目事实）

思考框架
└─ knowledge/thinking/ 中的开放问题（不提供项目答案）
```

`visualInspection.verified` 只有在记录数量覆盖全部图片时才有效。文件名、OCR、尺寸和元数据可以辅助发现素材，但不能证明画面的构图、材质、摄影或工艺。

## 输出约束

每次运行只生成：

1. `01-项目分析报告.md`
2. `02-Creative-Brief.md`
3. `03-Knowledge-Review.md`
4. `04-Design-Review.md`

系统不生成生图任务包、图片队列、数量计划、画幅比例或单张 Prompt。旧版参数 `fast`、`review`、`research` 仅为兼容入口，都会映射到同一个 `brief` 工作流。

## 写入边界

- 只写目标 `outputs/` 中的四份报告，以及显式 `--debug` 时的 JSON。
- Thinking Framework、模板与规则在运行时只读。
- 不写成长历史，不改正式 Knowledge，不执行 Git。
