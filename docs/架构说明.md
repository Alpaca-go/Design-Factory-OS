# 架构说明

Masterpiece-OS v3.2 的定位是 **AI Creative Brief Generator**。核心链路只有一条：

```text
Visuals → Brand Lock → Original Intent → Industry Benchmark → Creative Decision → Approved Brand DNA → Creative Brief
```

## 模块

1. `project-initializer`：创建标准项目目录，并安全整理根目录素材。
2. `inventory` / `parsers`：盘点图片、文档、压缩包与演示文件的可解析元数据。
3. `analyze`：建立 Brand Lock，并组织项目类型、行业与同类对标语境。
4. `brand-dna-decision`：强制执行 Original Intent、Industry Benchmark、Creative Decision 与显式批准，阻止视觉方案直接成为 Brand DNA。
5. `creative-reasoning`：把批准后的 Brand DNA 与其余项目推理转换为十部分 Creative Brief 数据契约。
6. `thinking-framework`：只读五类设计思考问题，并生成项目需要继续回答的问题。
7. `brief-review`：检查 Brief 的证据、决策链和进入创意发展的准备度。
8. `report`：固定渲染四份编号 Markdown；调试模式可额外写入 JSON。
9. `pipeline`：协调上述模块，不执行图片规划、Knowledge 写回、成长历史或 Git 操作。

## Brand DNA Decision 数据契约

```text
originalIntent
├─ statement
└─ evidence

industryBenchmark
├─ observations
├─ opportunities
└─ references（至少三个）

creativeDecision
├─ statement
├─ rationale
└─ tradeoffs

approvedBrandDNA（九个维度）
└─ approval.status = approved
```

只有四阶段内容全部完整且存在显式批准时，`approvedBrandDNA` 才会进入 Creative Brief。旧 `creativeReasoning.visualDNA` 会被保留为 `candidateBrandDNA` 供人工迁移，但输出保持待确认，避免把用户视觉方案直接包装成品牌结论。

## Creative Brief 数据契约

```text
brandIdentity
brandPositioning
designLanguage
emotionalDirection
approvedBrandDNA
photographyDirection
designRisks
mustKeep
canExplore
designGoal
```

除 Approved Brand DNA 外，其余字段可从项目配置、Brand Lock、Benchmark 或已记录的视觉核验事实中安全回退；无法确认时保留“待确认”。Approved Brand DNA 禁止回退生成。

## 证据分层

```text
项目事实
├─ Original Intent 及其依据
├─ 实际素材与逐张视觉核验
├─ 项目配置中的人工确认与批准记录
└─ 可追溯的公开来源

参考语境
└─ 同类 Benchmark（不得替代项目事实）

思考框架
└─ knowledge/thinking/ 中的开放问题（不提供项目答案）
```

`visualInspection.verified` 只有在记录数量覆盖全部图片时才有效。文件名、OCR、尺寸和元数据可以辅助发现素材，但不能证明画面的构图、材质、摄影或工艺。

## 输出约束

每次运行只生成：

1. `01-项目分析报告.md`
2. `02-Creative-Brief.md`
3. `03-Knowledge-Review.md`
4. `04-Design-Review.md`

系统不生成生图任务包、图片队列、数量计划、画幅比例或单张 Prompt。旧版参数 `fast`、`review`、`research` 仅为兼容入口，都会映射到同一个 `brief` 工作流。

## GPT Collaboration

交接输入固定为“已核验视觉方案 + `02-Creative-Brief.md`”。Masterpiece-OS 不输出图片规划；GPT 或创意团队在 Approved Brand DNA、Must Keep、Can Explore 与 Design Risks 的边界内，自主完成创意、图片规划与生成。

## Architecture Validation

3.2 的验证对象是 Creative Brief 能否替代旧生图任务包，不是增加新的项目输出。对九州美学、名济堂、香辣虾等本地真实项目采用同素材、同模型、同评审人的 A/B 测试：

| 指标 | 记录方式 |
|---|---|
| 总耗时 | 从输入交接到可接受首套结果的分钟数 |
| 首图质量 | 盲评 1–5 分，先展示第一张结果 |
| 满意度 | 品牌方或 Creative Director 1–5 分 |
| 是否返工 | 首轮后是否需要重做核心方向 |
| 品牌一致性 | 对 Approved Brand DNA 与 Must Keep 的盲评 1–5 分 |

A 组使用旧生图任务包，B 组只使用已核验视觉方案与 Creative Brief。真实项目记录留在本地，不进入仓库；系统仍只生成四份固定 Markdown。只有 B 组在不降低首图质量、满意度和品牌一致性的前提下减少总耗时或返工，才支持 Sprint 成功结论。

## 写入边界

- 只写目标 `outputs/` 中的四份报告，以及显式 `--debug` 时的 JSON。
- Thinking Framework、模板与规则在运行时只读。
- 不写成长历史，不改正式 Knowledge，不执行 Git。
